## 二、智能职业诊断模块详细规范

### 2.1 输入部分

| 字段 | 类型 | 必填 |
|---|---|---|
| 简历文件 | PDF/Word | 必填 |
| 最高学历 | 下拉 | 必填 |
| 工作经验年限 | 下拉 | 必填 |
| 目标岗位 | 文本+自动补全 | 必填 |

### 2.2 访谈

#### (a) 量表 12 道，3 维度 × 4 题

- 就业能力（4）
- 就业条件（4）
- 就业心态（4）

5 档（5/4/3/2/1），各维度满分 20。三档评估：强 ≥15 / 一般 10-14 / 弱 ≤9。**题目终稿你拟**。

#### (b) AI 访谈 4 题

- 简历澄清/补充 2 题（基于上传简历追问）
- 就业想法 2 题（3 年目标 / 不能接受的工作）

复用 career-report `/api/interview/*`。

### 2.3 报告 5 章

| 章节 | 数据来源 | 复用 |
|---|---|---|
| 第一章 总评 | 量表 → 3 维度 × 3 档 | **自写**（~1 天） |
| 第二章 优势能力与职业性格 | 量表 + AI 访谈 | career-report `PositionInfo`（雷达图直接搬） |
| 第三章 就业岗位建议 | 简历 + 目标岗位 + 量表 + 访谈 | career-report `Salary` + `PositionInfo` |
| **第四章 简历快诊** | 简历 + 目标岗位 | **career-report 第四章 ResumeDiagnosis 直接复用** |
| 第五章 行动建议 + 免责声明 | 全部前文 → 30/60/90 天计划 | **自写**（career-report 无对应章节） |

### 2.4 5 级帮扶标签（仅工作人员后台可见）

诊断完成时 AI 自动生成标签建议；标签**不展示给市民**，只在 `/admin/citizens` 后台显示。工作人员可手动调整 + 留痕。

| 标签 | 评估逻辑 | 服务策略 |
|---|---|---|
| 易帮扶 | 3 维度全强 / 2 强 1 一般 | 推岗位 + 轻指导 |
| 较难帮扶 | 1 强 2 一般 / 全一般 | 培训 + 月度跟进 |
| 难帮扶 | 1 弱 2 一般 / 2 弱 1 一般 | 重点辅导 + 双周跟进 |
| 重点帮扶 | 2 弱 1 一般 + 简历短板严重 | 一对一规划师 |
| 托底帮扶 | 3 弱 / 长期失业 / 健康约束 | 公益岗位 / 兜底政策 |

### 2.5 用户简档 + 服务跟踪（**仅工作人员后台**）

> **重要前提**：这块功能在 `/admin` 后台路径下，**市民端 (`/diagnosis/*`) 完全不可见**。市民只能看到自己的诊断报告，不知道自己被打了什么标签、有多少服务记录。

核心数据结构和功能：
- citizen_profiles / diagnosis_records / service_logs 三表（详见第五节架构图数据层）
- 工作人员后台：市民列表（按标签筛）/ 详情（简档 + 历史 + 时间线）/ 数据看板

## 三、智能创业诊断模块详细规范

### 3.1 输入部分

| 字段 | 类型 | 必填 |
|---|---|---|
| 项目名称 | 文本 | 必填 |
| 启动资金 | 数值（区间下拉：<10 万 / 10-50 万 / 50-200 万 / 200 万+） | 必填 |
| 主要产品/服务 | 文本（多行） | 必填 |
| 个人履历 | 文本（多行）或简历 PDF 上传 | 必填 |

### 3.2 访谈：纯 AI 多轮对话（与职业诊断不同，无量表）

#### 6 个核心维度
| # | 维度 | 示例问题 |
|---|---|---|
| 1 | 需求与定位 | 您的项目/产品具体解决了什么人的什么痛点？ |
| 2 | 产品与商业模式 | 目前的核心产品/服务是什么？商业模式和盈利方式是怎样的？ |
| 3 | 市场与竞争力 | 市场有多大？主要竞争对手是谁？核心优势在哪里？ |
| 4 | 团队能力 | 目前团队结构如何？核心成员有哪些关键资源和能力？ |
| 5 | 项目发展现状 | 项目现在处于什么阶段？已取得哪些成果（数据/客户/订单）？ |
| 6 | 发展规划与风险 | 未来 12 个月的发展目标是什么？目前最大困难和需求是什么？ |

#### 访谈规则
- **6 个核心问题**依次问完（占 6 轮）
- 每个问题如果回答不清晰/过短，**追问最多 1 次**
- **整体访谈轮数上限 8 轮**——意味着最多有 2 个问题被追问

#### 实现方式（V8：状态机由代码控制，LLM 只回布尔 — M9）
复用 career-report 的 `/api/interview/question` + `/api/interview/summarize` 流程，但**关键控制逻辑由代码而不是 LLM 决定**：

```ts
// 代码控制状态机（不漂移）
const QUESTIONS = [Q1, Q2, Q3, Q4, Q5, Q6];   // 6 维度核心问题写死
let currentIdx = 0;
let followupUsed = 0;
const MAX_TURNS = 8;

while (turn < MAX_TURNS && currentIdx < QUESTIONS.length) {
  ask(QUESTIONS[currentIdx]);
  const userAnswer = await getAnswer();

  // LLM 只回答两件结构化事
  const { sufficient, followupPrompt } = await llm.judge({
    response_format: 'json_object',
    schema: { sufficient: 'boolean', followupPrompt: 'string|null' }
  });

  if (sufficient || followupUsed >= 2) {
    currentIdx++;       // 下一个核心问题（代码决定）
  } else {
    ask(followupPrompt); // 追问 1 次
    followupUsed++;
  }
}
```

**关键约束**：
- LLM **不决定下一步问什么**，只判断"是否充分"+ 给追问 prompt
- 8 轮上限是代码硬约束（计数器溢出退出循环）
- LLM 漂移最多影响"该不该追问"的细节判断，不会破坏整体流程

### 3.3 报告（按你的明确：3 个章节）

| 章节 | 内容 | 复用 / 实现 |
|---|---|---|
| **第一章 综合评估** | 6 维度总分 + 整体定级 + 一句话画像（"具备初步创业条件，但市场定位需进一步聚焦"之类） | 自写评分 + LLM 生成画像描述（~1 天） |
| **第二章 项目概览** | 用户输入的 4 项（项目名/启动资金/产品/履历）+ AI 访谈摘要的清晰排版展示，让工作人员/市民一眼看懂"这是个什么项目" | 自写排版组件（~半天，纯展示） |
| **第三章 6 维创业评估** | 6 维度雷达图 + 每个维度的：得分 / 优势 / 风险 / 建议 | 雷达图复用 career-report `PositionInfo` 组件；逐维度内容 LLM 生成（章节并发） |

#### 6 维度雷达图
直接复用 career-report `PositionInfo` 章节的雷达图组件，把维度轴改成：需求/产品/市场/团队/现状/规划。

> **报告范围明确**：暂只这 3 章。"行动建议"和"政策匹配"等扩展内容**不在本期范围内**——如未来甲方要加，走变更签字。这一条要写进 SOW，否则被默认"应该有"。

#### 下载与预约按钮（本期前台不做）

- 市民端报告页**只显示报告内容**，不放下载/预约入口
- 这两个功能（连同其他业务功能）后续在"另一个管理后台"二期单独做
- 上海黄浦区的电话（63011095 / 63137613）和地址（中山南一路 555 号）**保留在合同 / 资料里待用**，本期不写进 UI

> 这意味着市民走完创业诊断后只能在线看报告。如果甲方后续提出"市民要能下载/转发/预约"，走变更签字。

### 3.4 5 级帮扶标签

创业诊断的标签是否沿用职业诊断的"易/较难/难/重点/托底"？还是用不同分类（例如：可创业 / 慎创业 / 不建议创业 / 转就业 / 兜底）？**这个你定**——V5 默认沿用同一套（5 级），便于工作人员看一个用户的总体画像。

## 四、政策与智能问答模块（V7 完整规范）

### 4.1 双页签结构

页面顶部两个 tab，切换不同知识库：

| 页签 | 知识库 | 文件量 | 用途 |
|---|---|---|---|
| **【政策与办事库】** | policy-kb | 3 个核心政策文件 | 政策解读、办事流程、补贴申领等 |
| **【创业与行业库】** | biz-kb | ?（**待你给出**） | 创业相关政策、行业资料、园区/孵化器信息 |

> ⚠️ **本周必问**：创业与行业库具体有几个文件？文件清单是什么？V7 暂按 5-10 个估。

### 4.2 回答策略（关键产品逻辑）

**核心原则：所有回答必须有真实出处 + 免责声明，不允许 AI 编造政策。**

实现方式：
- **第一优先级（命中知识库）**：用户问题命中 wiki 中已有内容 → DeepSeek 基于 wiki 答 + 引用 wiki 页 + 标注原始政策出处链接
- **第二优先级（部分命中）**：wiki 内容不足以完整回答 → DeepSeek 按 wiki 中已有的 + 显式声明"以下信息可能不完整，请咨询窗口" + 仅引用 wiki 中真实存在的链接
- **第三优先级（未命中）**：wiki 完全不覆盖 → 不让 LLM 自由发挥，回复"未在政策库中找到对应规定，请咨询黄浦区社保经办大厅或致电 63011095、63137613"
- **绝不允许**：AI 编造不存在的政策条款 / 编造虚假链接 / 用过期政策回答

System prompt 中硬性约束 + 输出后做"链接合法性校验"（regex 校验 wiki 内已注册的链接白名单）。

### 4.3 输出限制（V8 增加互斥兜底 — M10）

- **单轮输出 ≤ 1000 字**（system prompt 约束 + 后处理截断兜底）
- 每条回答末尾必须带：
  - 引用的政策文件名 + 段落号 / 公众号链接
  - 标准免责声明："本回答由 AI 生成，仅供参考，最终以人社部门解释为准"

**约束冲突防护（V8 新增）**：DeepSeek 在中文短输出时偶尔会丢链接（"必须带链接"和"≤1000 字"两条 system prompt 优先级冲突时模型倾向砍链接保字数）。后处理流程：
1. 链接合法性校验：检测输出是否有 wiki 内已注册的链接（白名单 regex）
2. 未带链接 → 重试一次，prompt 中强调"链接是硬约束，宁短勿丢"
3. 仍无链接 → 替换为兜底文案"未在政策库中找到对应规定，请咨询黄浦区社保经办大厅或致电 63011095、63137613"

**Prompt injection 防护**：
- 用户输入做敏感词过滤（黑名单：忽略、ignore、override、system 等指令词）
- 用户输入夹在 `<user_query>...</user_query>` XML tag 里送进 prompt，明确边界

### 4.4 3 个热点问题（页面预设）

页面醒目位置展示 3 个一键问按钮，点击后**直接展示预设答案**（不调 LLM，避免每次结果不一致）：

| Q# | 问题 | 答案来源 | 处理 |
|---|---|---|---|
| Q1 | 创业缺资金，有什么融资政策支持吗？ | https://mp.weixin.qq.com/s/J5GHjHKBw7_kA6nKH8FPVA | 抓取正文，**去掉引言和"各区联系方式"** |
| Q2 | 黄浦区有哪些办公场地可供租赁？ | https://mp.weixin.qq.com/s/HzZFJGXS7oQPlpcRwuCscA | 抓取正文，**去掉"极社空间介绍"段落** |
| Q3 | "黄浦创卡"是什么？ | **待你提供答案来源** | 同上格式 |

实现方式：
- 一次性人工抓取并清洗这 3 篇微信公众号文章 → 存为 wiki 中的预设回答
- **V8 新增（M13）原文存档**：抓取后存全文 markdown + 截图 PNG 存腾讯云 COS（防原文删除/失效）
- 用户点击热点问题 → 直接从数据库读预设答案 → 不调 LLM
- 在答案末尾加"了解更多请输入您的具体问题继续询问"
- **V8 新增 UI 容错**：链接挂了/原文 404 → 显示"原文已失效，以下为存档版本（[日期]抓取）"
- **V8 新增合同条款**：SOW 写明"热点问题答案内容由甲方授权使用"（避免版权风险落到乙方）

## 七·五、模块开发优先级分析（先做什么，为什么）

### 最终顺序（V8 决定）

```
①  政策与智能问答  →  ②  智能职业诊断  →  ③  CRM 子系统  →  ④  智能创业诊断
        W2                 W3-W4               W5                 W6
```

### 为什么是这个顺序

#### ① 政策问答先（W2，最先动）
- **业务链路最短**：用户问 → 检索 wiki → DeepSeek 答 → 显示。无多轮访谈、无结构化报告、无 PDF 导出
- **风险前置验证**：LLM Wiki 编译质量是 V8 最大的未知变量，**必须最早验证**（W0 试跑 + W2 真上线）。万一编译质量差，W2 切 LangChain RAG（+1-2 周）影响最小
- **甲方信心建立**：W2 末出第一个真实可用 demo，让甲方看到"AI 能用"，为后续两个月开发争取耐心
- **学习曲线**：单兵 Vibe Coding 从最简单的模块开始爬，避免 W3 直接啃硬骨头翻车

#### ② 职业诊断次之（W3-W4，核心模块）
- **复用率最高**：career-report 是该模块的 90% 基座，fork + 改 prompt 即可
- **铺路效应**：跑通"量表 + AI 访谈 + 多章节并发报告"完整引擎，**为创业诊断（W6）打地基**——创业诊断只是换 prompt + 换章节模板
- **CRM 的输入依赖**：5 级帮扶标签由职业诊断生成，CRM（W5）必须有这个输入才能展示价值
- **业务价值锚点**：市民端第一个体验闭环（输入→量表→访谈→报告），W4 末就能给甲方看完整流程

#### ③ CRM 第三（W5，价值放大器）
- **强依赖职业诊断输出**：5 级标签 + 诊断记录是 CRM 的核心数据源，必须等 W3-W4 跑通
- **甲方 daily driver**：工作人员真正每天用的就是这个模块——给甲方看到"系统真的在帮我们工作"的价值锚点
- **审计层已就绪**：W1 已建 audit_logs / llm_call_logs / consent_records 三张审计表，W5 只补业务 UI
- **复杂度可控**：纯 CRUD + 列表 + 详情 + 时间线，无 AI 工程，单兵 1 周拿下

#### ④ 创业诊断最后（W6，组装而非创造）
- **复用职业诊断引擎**：访谈框架、报告并发框架、雷达图组件、PDF 导出全继承
- **复用政策 wiki**：报告内"政策匹配"那块直接调 biz-kb（W2 已建）
- **报告章节最少**（仅 3 章，比职业 5 章少），且没有简历快诊这种重组件
- **状态机风险**：6 维度访谈 + 8 轮上限 + 1 次追问，逻辑稍复杂但 V8（M9）已改代码控制状态机，LLM 只回布尔，风险显著降低

### CEO 备选顺序（已记录但未采纳）

CEO 推荐：政策 → **CRM 优先（职业前）** → 职业 → 创业
- **CEO 论点**：CRM 是甲方 daily driver，应优先 UAT 反馈，避免后期被打回；市民诊断"使用率可能低于预期"
- **你的判断**：保持 V7 顺序。**理由**：CRM 的核心输入（5 级帮扶标签）依赖职业诊断生成，反过来做 CRM 拿不到真实数据，只能用假数据演示，意义有限
- **若执行中发现 CEO 论点成真**（比如 W4 末甲方说"市民端没人用"），回头切顺序仍来得及

### 每个模块的 "V0 demo 验收标准"（与第十二节验证方式对应）

| 模块 | V0 标准 | 不达标怎么办 |
|---|---|---|
| ① 政策问答（W2） | LLM eval 50 题准确率 ≥ 80% / 3 热点预设答案展示 / 双页签切换 / 1000 字+引用+免责 | 编译切 LangChain RAG，准确率不够调 prompt + 加 retrieval |
| ② 职业诊断（W4） | 完整链路打通 / HR 直觉评分 ≥ 7/10 / 5 级标签分布合理 | W3 末调阈值；prompt 调到模拟用户报告满意为止 |
| ③ CRM（W5） | 市民列表/详情/服务记录/数据看板四件套 / audit_logs 留痕生效 | 单纯 CRUD 不应该卡住；卡了说明前面架构有坑，回头修 |
| ④ 创业诊断（W6） | 6 维度访谈最多 8 轮（代码状态机）/ 3 章报告 / 雷达图 | 报告章节出不来就退到 stub 文案，雷达图复用 W4 组件 |

### 横向依赖关系（哪些模块共享"地基"）

```
W0 ICP 备案 + LLM Wiki 试跑
  │
W1 脚手架 + 多供应商 LLM + 审计层 + middleware + PIPL stub
  │  ▼ 所有模块都依赖这一层
  ├──▶ ② 职业诊断 ──▶ ③ CRM（依赖 5 级标签数据）
  │       │
  │       └──▶ ④ 创业诊断（复用访谈+报告引擎）
  │
  └──▶ ① 政策问答（独立模块，但 wiki 也供创业诊断"政策匹配"用）
```

**结论**：W1 的工程基建是所有模块的共同地基，**绝不能省**。这也是 V8 把 PIPL/审计/加密/middleware 全部前置到 W1 的根本原因。
