## 结论

**W0 风险已消除**：DeepSeek + 改造 prompt + MinerU PDF 解析的 LLM Wiki 链路，在 3 份真实政策素材上 D1 + D2 双项满分。**W2 政策问答上 LLM Wiki 路线可继续**，不切 RAG fallback。

### 这次 PoC 验证到的事

1. **DeepSeek 能严格按 prompt 约束做"不编造"政策内容**——所有 D1 抽样都是直接 quote 原文措辞，没有自由发挥
2. **引用回链 100% 可靠**——D2 抽样的 5 条 file:line 引用全部对得上 source 真实内容
3. **MinerU Agent API 免费档对中文政策 PDF 有效**——金额/电话/地址/比例数字 全部正确解析（pdftotext 在同一份 PDF 上把数字渲染成 ���）

### 这次 PoC **没**测到、留 W2 上线时观察的事

1. **跨主题分类**：DeepSeek 把 3 份都归到一个主题。W2 真上线 30+ 文件时，主题分类质量是另一项 unknown
2. **大文档分片处理**：本次 PDF 被切到 6 页（8.5MB / 12 页中前半段）才进 MinerU 免费档。W2 production 用 MinerU Precision API（200MB / 200 页）能直接全量。生产无阻塞
3. **回归性**：单次跑 100 分不代表稳定。W2 LLM eval suite 50 题会持续跑（CI 集成）
4. **创业政策外的 policy domain**（失业保险/社保办理）：本次素材都聚焦创业，未覆盖其他子域。W2 拿到甲方所有政策 PDF 后，前 5 份建议跑同一脚本快速 sanity check 再展开

### 顺手发现的 W2 改进点

- 主题文章里 DeepSeek 偶尔会写"原文未提供具体的申请步骤和材料清单，建议咨询..."。这种"承认不知道"的行为是好事（不编），W2 应保留 prompt 里的相关约束
- ji-she-kong-jian 这种"政策科普 + 真实地址"混合 source，DeepSeek 能合理只引用政策段而忽略 10 家具体基地的地址清单。结构化 article_sections 起到了过滤作用
